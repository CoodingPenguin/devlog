## 웹 서버 vs 웹 어플리케이션 서버

웹 서버는 웹 클라이언트의 요청을 받아서 처리하는 서버를 통칭하여 말한다. 하지만 다음과 같이 **웹 서버**와 **웹 어플리케이션 서버**로 세분화할 수 있다.

- **웹 서버** : 웹 클라이언트의 요청을 받아서 요청을 처리하고 그 결과를 웹 클라이언트에게 응답하는 서버
    - 주로 **정적 페이지**인 HTML, 이미지, CSS, JS 파일로 응답할 때 사용된다.
    - `ex` Apache httpd, Nginx, lighttpd, ISS 등
- **웹 어플리케이션 서버** : 웹 서버로부터 동적 페이지의 요청을 받아서 요청을 처리하고 그 결과를 웹 서버로 반환하는 서버
    - 주로 **동적 페이지** 생성을 위한 프로그램 실행과 데이터베이스 연동 기능을 처리한다.
    - `ex` Apache Tomcat, JBoss, WebLogic, WebSphere, Jetty, Jeus, mod_wsgi 등

## 정적 페이지 vs 동적 페이지

정적페이지와 동적페이지를 나누는 기준은 **사용자가 페이지를 요청하는 시점에 페이지의 내용이 유지 or 변경되는지**이다.

- `정적페이지` : 누가, 언제 요구하더라도 항상 같은 내용을 표시하는 웹 페이지
- `동적페이지` : 동일한 리소스의 요청이더라도 누가, 언제, 어떻게 요구했는지에 따라 각각 다른 내용이 반환되는 페이지
    - 동적페이지에는 프로그래밍 코드가 포함이 되어 있어서 페이지 요청 시점에 HTML을 생성해낸다.

## CGI 방식의 단점

CGI 자체는 **웹 서버와 독립적인 프로그램 사이에 정보를 주고받는 규격**을 말한다. 

전통적인 CGI 방식은 웹 서버가 여러 프로그래밍 언어로 만들어진 CGI 프로그램을 직접 호출하여 개별 프로세스를 생성하는 방식이다.

이럴 경우 각 요청마다 프로세스를 생성하므로 부하(Overhead)를 줄 수 있다.

## CGI 방식의 대안 기술

CGI 방식의 단점인 오버헤드를 줄이는 대안으로 두 가지 방식이 있다.

- `방법 1` **별도의 애플리케이션**을 스크립트 언어로 작성하고, 스크립트를 처리하는 **스크립트 엔진(인터프리터)**를 웹 서버에 내장시키는 방식이다. 파이썬에서는 *mod_wsgi 모듈*을 사용하고 있다.
- `방법 2` 애플리케이션을 처리하는 프로세스를 미리 **데몬**으로 가동시켜 놓은 후 웹 서버의 요청을 **데몬에서 처리**하는 방식이다. 이 방식에도 *mod_wsgi 모듈*을 사용한다.

<br>

후에 객체 지향 기술이 반영되면서 애플리케이션 전용 데몬인 **애플리케이션 서버 방식**으로 발전했다. 

- *JSP(Java Server Page), ASP(Active Server Page)*에서 웹 어플리케이션 서버 방식을 이용한다.
- 파이썬의 경우 *mod_wsgi, uwsgi, gunicorn 프로그램*이 웹 애플리케이션 서버이다.

## 애플리케이션 서버 방식

웹 애플리케이션 서버 방식은 **웹 애플리케이션 서버를 통해 간접적으로 웹 애플리케이션 프로그램을 실행**한다.

- 웹 서버는 `정적페이지`를, 웹 애플리케이션 서버는 `동적페이지`만 처리하도록 역할을 분담한다.
- 웹 서버는 정적페이지 처리 외에 캐시 기능, 프록시 기능, 클라이언트 수의 제한 및 처리 프로세스 관리, 요청 및 응답에 대한 로그 기록 등 **여러가지 기능**을 제공한다.
- 웹 애플리케이션 서버는 웹 서버보다 기능이 더 추가된고 종류도 다양해지고 있다. 다만 성능과 안정성 측면에서 개발용으로만 제한적으로 사용되고 있다.

## 웹 서버와의 역할 구분

하드웨어에서는 웹 서버를 `웹 서버 박스`, 웹 애플리케이션 서버를 `웹 애플리케이션 서버 박스`라고 부른다. 

- 모든 웹 사이트가 정적과 동적페이지를 같이 제공하므로 이 두 개의 서버를 동일한 HW 박스에서 가동시키기도 한다.
- 하지만 박스를 분리하는 것이 메모리 효율을 더 높일 수 있으므로 대형 웹 사이트는 둘을 분리하는 것이 보통이다.
